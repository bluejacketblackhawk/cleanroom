# ADR-007: CLI

- Status: Accepted (from the build handoff)
- Date: 2026-07-13
- Source: handoff/02-ARCHITECTURE.md § CLI (ADR-007)

## Context

ANVIL's core is a library (`anvil-core`), and the Tauri UI is a remote control. To enable headless batch processing, QA automation, and customer integration pipelines, a CLI binary (`anvil`) exposes the same processing graph without any UI. The CLI is the reference for testing and the integration surface for CI.

## Decision

`anvil` binary: JSON-first output, stable exit codes — the automation surface and QA harness engine.

**Command surface:**

```
anvil analyze in.wav --json
  → emit analysis results (loudness, peaks, voice/music, noise floor, etc.) as JSON stdout
  
anvil master in.wav --preset podcast-stereo -o out/ [--tier fast|standard|studio] [--report out/report.html]
  → render mastered audio file to out/, optionally with a visual report (before/after plots)
  
anvil transcribe in.wav --model small --srt --vtt --diarize
  → emit SRT/VTT subtitle files and JSON diarization labels
  
anvil batch ./drop --preset show-a --watch
  → watch a directory for new files, process each with preset, output to sibling directory
  
anvil models list|pull <pack>
  → list installed models or download a model pack by name
```

**Exit codes:** 0 = success; 1 = usage/config error; 2 = processing error (recoverable); 3 = system error (OOM, I/O failure); 4 = interrupted by user. Stable across releases.

**JSON output:** All structured output (analysis, transcripts, metadata, errors) as JSON. Machine-parseable, CLI tools (jq, etc.) can manipulate. Errors include `{error: "...", code: N, details: {...}}`.

## Consequences

**Enables:** Headless batch processing (for production pods, archival pipelines); CI automation and regression testing; third-party integrations; feature parity verification (CLI results must match UI results); reproducible eval corpus generation

**Constrains:**
- Enforce: CLI and UI use identical `anvil-core` processing graph (no code duplication). A feature missing from the CLI is a bug.
- JSON schema versioned; CLI version must output schema version in `--version` output. Schema changes are additive (backward-compatible) or require major version bump.
- Exit code stability: once defined, exit codes are API (cannot change meaning without major version).
- Error messages in JSON must be stable and machine-readable; user-facing explanations go in docs, not CLI output format.
